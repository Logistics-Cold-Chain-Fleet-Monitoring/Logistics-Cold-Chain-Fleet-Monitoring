# Task D3 — 10-Stage Edge ML Pipeline Mapping

This document maps LogiEdge explicitly to the 10 stages of the Edge ML pipeline from Module 4.

| Stage | Pipeline Stage | LogiEdge Implementation |
|-------|---------------|------------------------|
| 1 | **Problem Framing** | FreightBridge needs real-time cold-chain anomaly detection on 85 refrigerated trucks. The problem is framed as a 3-class classification task (Normal/Warning/Critical) based on temperature drift, vibration anomaly, and combined failure signatures. The 90-second detection SLA and rural connectivity gaps make edge inference mandatory — cloud-only is not acceptable. |
| 2 | **Data Collection** | Three sensor streams are collected on-truck: temperature at 1 Hz from a cargo probe (N(4.0°C, 0.3) normal), vibration RMS at 0.5 Hz from a compressor accelerometer (N(0.45g, 0.05) normal), and discrete door open/close events. The `simulator.py` script emulates these streams with configurable anomaly injection modes (`--anomaly {none|temp_drift|vibration|combined}`) and publishes to the local Mosquitto MQTT broker on the Raspberry Pi 5. |
| 3 | **Data Preprocessing** | Raw sensor data passes through a 5-sample moving average filter to reduce noise, then a 30-second sliding window (10-second step) extracts 6 features per window: temperature mean, temperature standard deviation, temperature rate-of-change (°C/min), vibration RMS, vibration peak amplitude, and vibration kurtosis. Features are z-score normalised using frozen statistics from `training_stats.npy` — never recomputed from live data. |
| 4 | **Model Training** | A 2-hidden-layer MLP (6→32→16→3 with ReLU activations and softmax output) is trained on ~294 labelled windows generated from 50 minutes of simulated data across all three classes. Training uses 80/20 stratified split, Adam optimiser, sparse categorical cross-entropy loss, and EarlyStopping (patience=15). The model achieves 93.22% validation accuracy, exceeding the 88% minimum threshold. |
| 5 | **Model Optimisation** | Three model variants are produced: M1 (FP32 baseline, 5.29 KB), M2 (Post-Training Quantisation to Full INT8 with 294 calibration samples, 4.59 KB), and M3 (35% structured pruning with PolynomialDecay schedule followed by INT8 PTQ, 4.52 KB). Benchmarking across 5 metrics (mean latency, p95 latency, file size, accuracy, energy) identifies M2 as the deployment recommendation based on Class 2 recall exceeding 95%. |
| 6 | **Model Conversion** | The trained Keras model is exported to SavedModel format using `model.export()`, then converted to TFLite FlatBuffer format using `tf.lite.TFLiteConverter.from_saved_model()`. For INT8 variants, the converter uses a representative dataset generator (294 samples) with `TFLITE_BUILTINS_INT8` target ops, producing models with int8 input/output types for maximum edge efficiency. |
| 7 | **Containerisation** | The inference service is packaged in a Docker container based on `python:3.11-slim`. The Dockerfile orders layers for OTA efficiency: `pip install` (large, rarely changes) appears before `COPY model.tflite` (small, changes every 6 weeks). The container accepts `MODEL_PATH` as an environment variable to switch between model variants without rebuilding. On rebuild with only a model change, only the final layer (~280 KB) is re-pushed, saving bandwidth across the 85-truck fleet. |
| 8 | **Deployment** | An Ansible playbook (`logibridge_deploy.yml`) with exactly 7 idempotent tasks handles fleet-scale deployment: create target directory, copy model, copy reference distribution, stop old container, pull updated image, start new container with environment variables, and verify health after 15 seconds. Running the playbook twice without changes produces `changed=0`, confirming idempotency for safe repeated deployment. |
| 9 | **Inference** | The `inference_service.py` subscribes to the MQTT feature topic (`logibridge/trucks/{truck_id}/features/window`), deserialises the 6-value feature vector, runs TFLite inference (handling both FP32 and INT8 quantisation automatically), and publishes the result (class label + confidence score) to `logibridge/trucks/{truck_id}/inference` (QoS 1). For Warning/Critical detections, an additional alert is published to `logibridge/trucks/{truck_id}/alerts/anomaly` (QoS 2 — exactly-once delivery for safety-critical alerts). |
| 10 | **Monitoring & Maintenance** | A PSI (Population Stability Index) drift monitor tracks the model's output confidence distribution against a reference built from 300 clean Normal-class windows. Every 60 seconds, it computes PSI over a rolling window of the last 100 inferences across 4 bins ([0, 0.25), [0.25, 0.50), [0.50, 0.75), [0.75, 1.0]). When PSI exceeds 0.25, a `[LOGIBRIDGE DRIFT ALERT]` is triggered, signalling that the model's behaviour has shifted — likely due to sensor degradation or environmental changes — and a model retrain cycle should be initiated. |
